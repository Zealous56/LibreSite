Bolte

The Bitcoin network has been around for nearly a decade now. Over the past few years, I have been following recent developments in distributed ledger technology. I decided it's time I write my own blockchain to better understand it, programatically speaking. Bolte is more a demo how blockchain technology works than a real implementation. As you can see in the TODO, there are many aspects of blockchain that I would eventually like to program. I plan on extending this project at a later date, but for now I will postpone further development as I have hit a roadblock. I need more practice in Python and in building multifaceted, networked applications.

I am happy to have accomplished a basic demo in a language I have had only a few days practice using. In Bolte, you can see examples of the structure of blockchain and blocks, transactions (without digital signatures), proof of work, and state in blockchain.py. Validation examples are in a separate module named validator.py.

LibreSite

What is LibreSite? You're looking at it! Well, sort of. LibreSite is the template for the site you are looking at. I wrote it myself. So in a way, there is a recursion since you are reading about the site template on the site itself. I wrote LibreSite for gaining experience with HTML and CSS. A central goal was also to create a website that respects the user visiting it. For example, not requiring the user to run foreign Javascript on their computer just to view the site. Also, by respecting the privacy of the user by downloading custom fonts locally instead of from Google Fonts. This way, Google isn't necessarily alerted who is visiting my site and when. And finally, I have made the code for generating the template to this site freely available for anyone to run, copy, modify, and share.

As we were learning pug in databases and web design class, I decided I would write LibreSite using it. I also used Node.js to compile the pug. All of this worked out well. What did not work out so well was trying to write my own CSS stylesheet. Lesson learned. I will never try that again. Spending hours debugging and trying to make a flexible layout is not worth the time. There are stylesheets online that take care of all that for you such as w3css from w3schools. Another thing that could be improved right now is the flexibility of the layout, meaning more options for easy user styling, as well as a less hacky content management system, which currently just reads from text files. Overall, I learned a lot and I'm still proud of what I've accomplished.

Numbers-to-Words

The project is self-explanatory. It converts numbers into words. For example, 65 becomes "sixty five". The algorithm was not copied. I thought it up on my own although I'm positive it's unoriginal. It was my first experience in both inventing an algorithm and then putting it into code. If you look at the first commit, you will notice how awful the code structure is. It even makes me want to cringe looking back on it! That code came from an old project written years ago that I wrote but never released. I committed the initial code to show from where I started, rather than rewriting the algorithm from scratch. It had many problems. Nearly all of the first 81 lines are unnecessary variable declarations. On line 144, there is a for loop that executed 3 times more often than needed. Lines 150-152 are try/catch blocks where the exception is immediately caught to prevent an arrayIndexOutOfBoundsException which should not happen. On top of that there is a reverseString() function which can be thrown out entirely. The variable declarations at the beginning also have meaningless names. The variables later create a problem that cascades down the rest of the program forcing parts of the code to be up to forty times longer than the optimized version I finished.

The project has thorough unit testing which actually helped me uncover bugs. In the optimized algorithm, I removed the excessive variables, fixed the loop, and replaced the try/catches.

Sorting-Algorithms

Simple sorting algorithms in C++. All my previous work was in Java, so I thought it was time to change things up. Most all the sorting algorithms I wrote need to swap two elements in an array. So there is a swap.cpp file separated from the rest of the cpp files that execute the algorithms. The implementations are not my own. This project was an exercise in learning the algorithm by writing it, not inventing my own implementation.

My favorite algorithms are recursive. Unfortunately recursion is memory inefficient and is often replaced in industry code by loop structures for that reason. My all time favorite is quicksort for its simplicity and flexibility. Because you are free to choose any pivot point, you can optimize quicksort to suit your dataset. Insertion and selection sort are very similar. Bubble sort is interesting since its swapping is more dynamic. Radix sort was my least favorite algorithm. I found it unintuitive and boring to program. Although I can't argue with the efficiency and might use it in a future project.

Diceware-GUI

I can't code just to code. I must have an end goal in mind and a strong purpose helps. For Diceware-GUI, the purpose was simply to create a graphical interface for the Diceware method of generating passwords. For those who don't know, Diceware was concocted by Arnold G. Reinhold. It is a method for generating secure, memorable passwords. Why not just use a password manager? There are certain scenarios where password managers are inappropriate. For example, what will the password to your password manager be? Generating a secure password for your manager and writing it down to remember it is no good. Someone may find it. Trying to remember it is impractical since it's a bunch of scrambled generated characters. What do you do? That's where Diceware comes in.

Simply roll some die as your source of randomness and match them with the numbers on the Diceware list. There is a one-to-one mapping from each number to a unique word. Generate enough words and concatenate them into one password, and lo and behold, you have a secure password. And more importantly, with a little effort, you can remember the password. Because what good is having a password you can't remember?

I would caution you not to make your own word list, since the letter frequency of the list is important. The lists found on the Diceware website have been crafted to be secure. They are available in many languages and I highly recommend reading through the original Diceware website for more detailed information.

My program is fairly simple. It uses a basic GUI and some regex magic to read out all the lists of any language. The Java SecureRandom class is utilized for generating random numbers. While a useful tool, I would still highly recommend generating passwords by hand with casino grade dice for added security, since your passphrase is only as secure as the source of randomness. If you use my program to generate passwords, keep in mind that the password is only as secure as the computer you generate them on. Also, my program is not thoroughly tested or audited by a third party, and I did not unit test it. So I would avoid using it for serious security purposes. I wouldn't rely on anyone else's programs for Diceware either unless they had been security audited. The best way to avoid this problem is to just use real dice.

RomanNumeral

Roman Numeral was my first practice in creating a pure function Java class. It has exception handling and uses abstract. It can convert an integer into a roman numeral string using toRoman() or roman numeral strings to integers using toRoman(). Also, it is well-documented. Rather than not writing enough comments, I have the opposite tendency. I tend to over comment my code. In this project, there is so little code and so much documentation, it seems like 40% of the program is just documentation.

Rational

My first project. I learned to use Github and make commits. There is a class for integer primitives. There is also a class that supports BigInteger. Both classes are well documented. I also unit tested them in FracTest.java. The rational objects are internally represented as a numerator and denominator. Representing the rational object as a float or double would be unworkable for several reasons. Rationals are operated on much the same way you would do the operations by hand, so reading the classes' code should be intuitive if you know basic Java and are familiar with rational numbers.
